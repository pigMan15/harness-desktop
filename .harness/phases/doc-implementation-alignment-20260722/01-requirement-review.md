# 需求评审

## 目标

核对当前项目整体实现与 `doc` 中设计文档的差异，输出差异清单，并完善可直接修复的文档或实现不一致。

## 范围

- 读取并提取 `doc/desktop-architecture.md` 与 `doc/desktop-implementation-plan.md` 中的设计承诺、交付阶段和验收标准。
- 对照当前仓库实现，包括 Runtime、Electron Desktop、Renderer、Contracts、Scripts、Tests、README 和配置。
- 输出差异清单到当前 `phase_dir`，包含：设计项、当前证据、差异类型、建议处理。
- 对低风险且可在仓库内直接完成的差异进行完善，例如文档状态、计划勾选状态、缺失说明、明显声明不一致或测试/配置小修。
- 运行相关验证命令，并将结果写入门禁和 evidence。

## 非目标

- 不重写 Harness Desktop 的总体架构。
- 不新增大范围产品功能，例如完整 E2E 套件、Windows VM 安装验收、自动更新服务或真实代码签名。
- 不伪造未执行的测试、安装验证或发布结论。
- 不将 `.harness` 协议升级为 v2。

## 验收标准

- [ ] 标准 1：差异清单存在且覆盖 doc 中主要架构/实施计划条目。
  - 验证方式：检查 `state.phase_dir` 中差异清单文件非空，并能追溯到 doc 与实现文件。
- [ ] 标准 2：已处理差异有明确文件变更，未处理差异有原因和剩余风险。
  - 验证方式：检查 `11-development.md`、`15-evidence.json` 和 git diff。
- [ ] 标准 3：如修改源码，相关 TypeScript/Python 测试或等价检查已运行并记录。
  - 验证方式：查看 `12-compile.md`、`13-unit-test.md`。
- [ ] 标准 4：文档不再把无法证明完成的事项表述为已完成。
  - 验证方式：复查 README/doc 中的完成声明与命令证据。

## 开放问题

- 当前无法在本地直接证明“干净 Windows VM 安装、升级、卸载”和完整 Playwright E2E 是否完成；若发现文档声称完成，只能降级为剩余差异或待验证项。
- 实施计划大量 checkbox 可能是历史计划未回填，也可能是真未完成；需要以当前代码和测试证据判断，不仅凭勾选状态。

## 风险备注

- 最大风险是把“已存在代码文件”误判为“首发验收已完成”。本次核对必须区分：实现存在、测试覆盖、端到端验收、发布证明四种不同证据强度。
