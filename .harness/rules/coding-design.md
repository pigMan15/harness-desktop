# 编码设计确认规则

目的：在真正编码前，先确认本次实现的编码思想、架构风格和关键设计选择，避免直接堆代码。

## 触发条件

以下场景必须生成编码设计并等待用户确认：

- `FEATURE / HIGH`
- `REFACTOR / MEDIUM`
- `REFACTOR / HIGH`
- `BUG_FIX / HIGH`
- 用户明确要求“先确认编码设计”“先给实现思路”“关注设计模式/架构风格”
- 当前实现会改变模块边界、核心业务流程、接口契约、权限模型、状态流转或数据一致性策略。

以下场景不强制等待确认：

- `FEATURE / MEDIUM` 的常规功能开发。
- 已确认方案内的细节实现、字段补充、样式调整、校验补充、文案调整。
- 不改变模块边界和架构风格的小范围代码修改。

这些场景可以在 `11-development.md` 中简要记录编码思路，不必单独生成 `10-coding-design.md` 并停等确认。

如果开发中发现原计划不适用，或需要扩大范围、调整架构、改变接口契约，应暂停并升级到编码设计确认。

## 必须产出

编码前生成：

```text
state.phase_dir/10-coding-design.md
```

内容必须包含：

- 推荐编码方案。
- 采用的整体架构风格。
- 是否使用设计模式，以及为什么。
- 模块边界和职责划分。
- 前后端协作点或接口契约。
- 复用现有代码的位置。
- 不采用的方案和原因。
- 中文注释策略。
- 风险点和回退策略。
- 需要用户确认或补充的问题。

## 用户确认

只有进入 `CODING_DESIGN_CONFIRMATION` 节点，或用户明确要求确认时，生成 `10-coding-design.md` 后才必须停止，等待用户二次确认或补充。

用户确认前，不要开始大范围代码修改。允许做只读代码检索、接口定位、依赖关系确认等准备工作。

## 输出模板

```markdown
# 编码设计确认

## 推荐方案

## 架构风格

## 设计模式选择

## 模块边界

## 前后端协作点

## 复用现有代码

## 不采用的方案

## 中文注释策略

## 风险与回退

## 需要用户确认
```

## 避免

- 不要写泛泛的“按现有架构实现”。
- 不要为了套设计模式而引入设计模式。
- 不要在未确认前做大范围重构。
- 不要忽略项目既有风格和现有代码复用点。
