# Knowledge Promotion Policy

run 结束后，不要把所有 phase 都写回知识库，也不要重复沉淀 PRD 已经明确的内容。

核心原则：

```text
只沉淀相对原始 PRD / context-pack 新增的工程知识。
```

## 值得沉淀

- PRD 没写清，但开发过程中确认下来的业务规则。
- PRD 与代码现状冲突，最终确认的处理结论。
- 实现过程中发现的业务不变量。
- 实际代码入口、模块边界、关键类和关键方法。
- 可复用实现模式。
- 重复出现的坑。
- 编译、测试、部署的稳定经验。
- 线上或预发问题根因。
- 架构决策及取舍。
- 验证中发现的限制、阻塞和可靠命令。

## 不值得沉淀

- PRD 已经明确写过，且开发中没有变化的内容。
- 原型截图里的普通页面说明。
- 一次性的临时日志。
- 已经过期的方案草稿。
- 与业务无关的工具输出。
- 未验证的猜测。
- 只对当前 run 有意义的细节。

## 增量判断

沉淀前必须回答：

```text
这条知识相对 PRD / context-pack 新增了什么？
下次类似需求是否能复用？
有没有证据支撑？
```

如果三者都答不上来，不要沉淀。

## 沉淀等级

```text
case      单次案例
pitfall   已确认的坑
pattern   可复用模式
rule      应进入 harness rules 的稳定约束
adr       架构决策
```

## 输出草稿格式

```markdown
# 知识沉淀草稿

- 类型: case | pitfall | pattern | rule | adr
- 标题:
- 来源 run:
- 关联 PRD:
- 相对 PRD 的新增点:
- 关联模块:
- 关联文件:
- 结论:
- 证据:
- 下次复用方式:
- 建议写入位置:
- 是否需要用户确认: yes
```
